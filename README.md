### 실시간 소켓 통신을 활용한 모던 채팅 애플리케이션 (React, Express, Socket.io)

2026.06 (1 人 팀 프로젝트)

## 📌 Summary

Socket.io를 활용해 지연 없는 실시간 양방향 1:1 채팅 및 접속 상태(온라인/오프라인) 표시 기능을 구현했습니다.

보안과 개발 생산성을 높이기 위해 Clerk을 도입하여 인증 시스템을 구축하고, Webhook을 이용해 자체 MongoDB 서버와 유저 데이터를 실시간으로 동기화했습니다.

UI는 최신 컴포넌트 라이브러리인 HeroUI와 Tailwind CSS를 결합하여 구성했으며, Context API를 활용해 라이트/다크 모드 및 다양한 컬러 프리셋을 지원하는 테마 시스템을 구축했습니다.

전역 상태 관리는 리렌더링 최적화에 유리한 Zustand를 도입하여, 타이핑이나 메시지 수신 등 상태 변화가 잦은 채팅 환경에서도 성능 저하 없이 부드러운 UX를 제공합니다.

백엔드는 Node.js와 Express 기반으로 RESTful API를 설계하고, Mongoose를 이용해 MongoDB와 연동하여 사용자 및 메시지 데이터를 효율적으로 관리합니다.

프론트엔드와 백엔드 간의 데이터 흐름을 명확히 하고, 타입 안정성을 보장하기 위해 프로젝트 전체에 TypeScript를 적용했습니다.

주요 기능 : Clerk 기반 회원가입 / 로그인, 실시간 1:1 채팅, 현재 접속 중인 유저 상태(온라인) 표시, 실시간 메시지 수신 알림, 커스텀 테마 및 프리셋 설정

## 🤔 Background

웹 개발의 꽃이라 불리는 '실시간 양방향 통신'을 깊이 있게 이해하고 싶었습니다.

일반적인 HTTP 요청(REST API)은 클라이언트가 요청할 때만 서버가 응답하기 때문에, 누군가 메시지를 보냈을 때 즉시 화면에 띄워야 하는 채팅 서비스에는 적합하지 않았습니다.

이를 해결하기 위해 WebSocket 기반의 Socket.io를 도입하여 서버가 먼저 클라이언트에게 데이터를 푸시할 수 있는 환경을 구축하고자 했습니다.

또한, 채팅 앱은 '메시지 입력', '스크롤 이동', '새 메시지 도착' 등 화면의 상태 변화가 매우 빈번하게 일어납니다.

기존에 사용하던 Context API로 이 모든 상태를 관리하면 불필요한 하위 컴포넌트까지 리렌더링되는 문제가 있어, 렌더링 최적화에 탁월한 Zustand를 실무 수준으로 적용해 보는 것을 목표로 삼았습니다.

마지막으로, 인증 로직을 직접 구현하는 대신 최근 각광받는 SaaS형 인증 솔루션인 Clerk을 도입하여, 모던 프론트엔드 생태계의 트렌드를 경험하고 인증 보안의 완성도를 높이고자 했습니다.

## 🔍 Meaning

프론트엔드와 백엔드, 그리고 서드파티(Clerk) 서버까지 3개의 축이 맞물려 돌아가는 아키텍처를 설계하며 많은 시행착오를 겪었고, 다음과 같은 값진 교훈을 얻었습니다.

상태 관리 도구의 적절한 분리 (Separation of Concerns):
거의 변하지 않는 '테마 상태'는 Context API를 사용하여 설계도(Definition)와 Provider를 분리하는 패턴으로 구현했습니다. 반면, 1초에도 수십 번씩 변할 수 있는 '채팅 메시지'와 '입력창 상태'는 Zustand를 사용해 필요한 컴포넌트만 선택적(Selector)으로 리렌더링되도록 구성하여 성능을 극대화했습니다.

웹훅(Webhook)을 통한 외부 서버와의 데이터 동기화:
Clerk 인증 서버에서 가입한 유저 정보를 로컬 MongoDB에 동기화하기 위해 웹훅을 구현했습니다. 이 과정에서 외부 서버(Clerk)가 로컬호스트(Localhost) 환경에 접근하지 못하는 네트워크 구조적 한계를 마주했습니다. render.com 서비스를 이용해 배포형 개발 환경을 구축하여 성공적으로 웹훅 이벤트를 수신하고 DB를 업데이트할 수 있었습니다.

실시간 통신(Socket.io)의 생명주기와 동기화 문제 해결:
채팅 중 한쪽 방향으로만 메시지가 가거나, 온라인 상태가 오락가락하는 '유령 소켓' 버그를 겪었습니다.
원인을 분석해 보니 두 가지 문제가 있었습니다.

고유 식별자(ID) 불일치: MongoDB가 생성한 \_id와 Clerk이 발급한 clerkId가 혼용되어 데이터가 엇갈리는 것을 발견하고, 시스템 전체의 통신 기준 ID를 clerkId로 단일화하여 데이터 정합성을 맞췄습니다.

React 생명주기 충돌: Strict Mode 환경에서 컴포넌트가 두 번 마운트되며 소켓이 중복 연결되는 문제를 파악했습니다. Zustand 스토어 내부에 소켓 객체의 존재 여부를 엄격하게 검사하는 방어 로직(Guard)과 확실한 클린업(Cleanup) 로직을 추가하여 중복 연결과 메모리 누수를 완벽히 차단했습니다.

## 🔨 Technology Stack(s)

TypeScript, React.js (Vite), Zustand, Tailwind CSS, HeroUI, Socket.io (Client/Server), Clerk (Auth & Webhook), Node.js, Express, MongoDB, Mongoose
